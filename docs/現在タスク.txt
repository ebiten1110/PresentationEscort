
Phase 5.10：顔検出方式の切り替え対応
Presentation EscortのPhase 5.10「顔検出方式の切り替え対応」を実装します。

現在はOpenCVのHaar Cascadeで顔を検出していますが、暗所・横顔・斜め顔・小さな顔の検出精度が不足しています。

今後、以下の検出方式を切り替えて比較できる構成にしたいです。

・Haar Cascade
・OpenCV YuNet
・将来的にはMediaPipe
・将来的にはInsightFaceまたはONNX形式の軽量モデル

動作環境は以下です。

・Raspberry Pi 3 A+
・Python
・OpenCV
・USB WebカメラまたはスマホIPカメラ
・GUIなしのヘッドレス環境
・検出結果は画像保存とターミナルログで確認
・処理負荷をできるだけ軽くする必要がある

以下を実装してください。

1. config.pyのFACE_DETECTOR_MODEで検出方式を切り替えられるようにする
2. FACE_DETECTOR_MODE = "haar" または "yunet" を選べるようにする
3. detector.pyを作り、検出方式の違いを追従処理側から隠す
4. どの検出方式でも、戻り値をx, y, w, h, confidenceの共通形式にする
5. 検出された複数の顔からメインターゲットを選ぶ共通処理を作る
6. モデルやCascadeが読み込めない場合のエラー処理を入れる
7. 処理時間を計測し、1フレームごとの推論時間をログ表示する
8. Haar版の既存処理を壊さずに残す
9. YuNetモデルが未配置の場合は、分かりやすいエラーを表示する
10. 将来的にMediaPipeやInsightFaceを追加しやすい設計にする

新しいフォルダ構成、config.pyの追加部分、detector.pyの全文、検出方式を確認するdetector_test.pyの全文を提示してください。

Raspberry Pi 3 A+で動かすため、不要なクラス階層や重い処理は避け、初心者でも理解できる構成にしてください。

実行手順、結果画像の確認方法、処理速度の見方、HaarとYuNetを切り替える方法も説明してください。

Phase 5.12：YuNet版追従ループ統合
Presentation EscortのPhase 5.12「YuNet版Advanced Follow Loop」を実装します。

Phase 5.11でOpenCV YuNetによる顔検出が成功している前提です。

現在の追従機能は以下です。

・顔の中心X座標からLEFT / RIGHT / STOPを判定
・顔の中心Y座標からHEAD_UP / HEAD_DOWN / HEAD_CENTERを判定
・顔の検出幅からFAR / GOOD / NEARを判定
・顔が中央かつFARの場合にFORWARD
・Raspberry PiからESP32へ115200bpsでシリアル送信
・同じコマンドの連続送信を抑制
・終了時にSTOPとHEAD_CENTERを送信

使用可能なESP32コマンドは以下です。

・LEFT
・RIGHT
・FORWARD
・STOP
・HEAD_UP
・HEAD_DOWN
・HEAD_CENTER

以下を実装してください。

1. Haar Cascadeの代わりにYuNetで顔を検出する
2. FACE_DETECTOR_MODEでHaarとYuNetを切り替えられるようにする
3. YuNetが返した顔情報をx, y, w, h, confidenceへ変換する
4. confidenceが低い検出結果を除外する
5. 複数人がいる場合は基本的に最も大きな顔を追従する
6. 前回追従していた顔に近い顔を優先する仕組みも追加する
7. 左右・上下・距離判定を連続処理する
8. LEFT / RIGHTで向きを合わせた後にFORWARDを実行する
9. NEARの場合は必ずSTOPする
10. 顔を見失った場合は一定回数待ってからSTOPする
11. 判定結果と推論時間を最新画像へ描画する
12. 最新画像をadvanced_follow_loop_latest.jpgとして保存する
13. Ctrl+C終了時や例外発生時にSTOPとHEAD_CENTERを送る
14. Raspberry Pi 3 A+で処理落ちしにくいようにする

config.pyの必要な設定、detector.pyの必要な修正、advanced_follow_loop.pyの完全なコードを提示してください。

コードをファイル単位で省略せず提示し、実行手順、調整値、ログの確認方法、Haarへ戻す方法も説明してください。

最後に、Phase 5.12用として「検出器接続」「座標判定」「シリアル送信」「連続追従」「Haar比較」「完了判定」の6区分に分けたチェックリストを必ず生成してください。
Phase 5.13：追従動作の安全化・安定化
Presentation EscortのPhase 5.13「追従動作の安全化・安定化」を実装します。

現在、Raspberry Piの顔検出結果からESP32へ以下の命令を送っています。

・LEFT
・RIGHT
・FORWARD
・STOP
・HEAD_UP
・HEAD_DOWN
・HEAD_CENTER

二足歩行ロボットなので、検出結果が一瞬変化しただけで命令を切り替えると、転倒・振動・意図しない前進が発生する可能性があります。

以下の安全機能を実装してください。

1. 同じ判定が指定回数連続した場合のみコマンドを確定する
2. LEFTとRIGHTを直接切り替えず、一度STOPを挟む
3. FORWARDを実行する前に顔が中央範囲内か確認する
4. FORWARDを連続実行できる最大回数を設定する
5. FORWARD実行後は一定時間STOPするクールダウンを追加する
6. 顔がNEARの場合は即座にSTOPする
7. 顔を見失った場合は短い猶予後にSTOPする
8. シリアル通信が切れた場合は安全終了する
9. カメラ取得に失敗した場合はSTOPを送る
10. Ctrl+C、例外、通常終了のすべてでSTOPとHEAD_CENTERを送る
11. ロボットの状態をIDLE / TURNING / MOVING_FORWARD / LOST_TARGET / EMERGENCY_STOPとして管理する
12. EMERGENCY_STOPを手動で有効化できる設定を作る
13. 1回の実行時間またはループ回数に上限を設定する
14. 状態遷移とコマンド送信理由をログへ出す

既存のadvanced_follow_loop.pyを安全化する形で、config.pyの追加設定とsafe_follow_loop.pyの完全なコードを提示してください。

安全機能を一つずつテストする方法と、実機テスト前にサーボ電源を切った状態で確認する手順も説明してください。

最後に、Phase 5.13用として「机上確認」「サーボ電源OFF確認」「低速実機確認」「異常時確認」「安全終了確認」「完了判定」の6区分に分けたチェックリストを必ず生成してください。

Phase 7：OLEDステータス表示
Presentation EscortのPhase 7「OLEDステータス表示」を実装します。

使用機器は以下です。

・ESP32-WROOM-32
・PCA9685
・WaveShare 0.96inch OLED Module
・PCA9685とOLEDはI2C接続予定
・ESP32のSDAはGPIO21
・ESP32のSCLはGPIO22
・PCA9685のI2Cアドレスは0x40
・OLEDは0x3Cまたは0x3Dの可能性がある

現在ESP32が管理している状態は以下です。

・歩行状態
・最後に受信したコマンド
・頭サーボの上下左右状態
・ライトON / OFF
・MANUAL / FOLLOWなどのモード
・シリアル通信状態

OLEDへ以下のような情報を表示したいです。

Presentation Escort
MODE: FOLLOW
FACE: FOUND
MOVE: FORWARD
HEAD: UP
LIGHT: ON

以下を実装してください。

1. I2CスキャナでOLEDのアドレスを確認する手順を作る
2. PCA9685とOLEDを同じI2Cバスへ並列接続する
3. PlatformIOのlib_depsに必要なOLEDライブラリを追加する
4. oled_display.hとoled_display.cppを作る
5. OLED初期化失敗時もロボット本体は動作を継続できるようにする
6. 状態が変化したときだけ画面を更新する
7. 更新頻度を制限し、歩行処理を妨げないようにする
8. FOLLOW、MANUAL、FACE FOUND、FACE LOSTなどを表示できるようにする
9. MOVE、HEAD、LIGHTの状態を表示する
10. main.cppとserial_receive.cppへ統合する
11. OLEDなしでもビルドできる設定を追加する
12. 既存のPCA9685制御を壊さないようにする

platformio.ini、config.h、oled_display.h、oled_display.cpp、main.cpp、serial_receive.cppの必要なコードをファイルごとに提示してください。
